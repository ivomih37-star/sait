# Безопасность и честные ограничения

Этот документ описывает, что этот проект реально даёт с точки зрения приватности и
безопасности, а что — нет. Читайте перед тем, как полагаться на него в чувствительных
сценариях.

## Что защищено

- **Трафик между вами и сервером** зашифрован WireGuard (ChaCha20-Poly1305, современная
  и хорошо проверенная криптография).
- **DNS-запросы** уходят через DNS-over-HTTPS (dnscrypt-proxy → Cloudflare/Quad9),
  провайдер не видит, какие домены вы резолвите.
- **Веб-панель wg-easy** не доступна из интернета — только через SSH-туннель
  (`ssh -L ...`), это резко снижает поверхность атаки.
- **На сервере не пишутся** access/error-логи Xray, query-логи dnscrypt-proxy,
  докер-логи ограничены по размеру и ротации.
- **fail2ban** банит IP после нескольких неудачных попыток SSH-логина.
- **Автообновления безопасности** снижают риск эксплуатации известных уязвимостей ОС.

## Чего этот проект НЕ гарантирует

### 1. Обход VPN-детекта у Netflix/Spotify/банков — best-effort, не гарантия

IP-адреса Hetzner/Vultr/Timeweb и других облачных провайдеров публично известны и
находятся в базах данных проверки "datacenter/proxy/VPN ASN" (используются Netflix,
Spotify, некоторыми банками). Ни один скрипт разворачивания сервера не может обойти
это технически — это репутация конкретного IP-адреса, а не свойство протокола.

Что реально помогает:
- Xray (VLESS+Reality) маскирует TLS-handshake под легитимный сайт (`XRAY_SNI`) —
  это обходит **DPI-блокировку по сигнатуре протокола**, а НЕ проверку репутации IP.
- Если конкретный IP уже "спалился" — единственное решение это смена IP
  (пересоздание сервера у провайдера, часто с новым IP) или использование
  дополнительного резидентского/мобильного прокси поверх VPN (вне рамок этого проекта).

### 2. Split-tunnel на телефоне/ПК — приближение, не точная маршрутизация по доменам

Протокол WireGuard и стандартные клиентские приложения (iOS/Android/Windows/macOS)
принимают маршрутизацию только по `AllowedIPs` (диапазоны IP), а не по доменным именам.
Поэтому для этих клиентов используется список "весь интернет минус Россия"
(`routing/world-minus-ru.txt`, пересчитывается `update-routes.sh`), а не точный список
"именно эти сервисы". Практические следствия:
- Иностранные CDN/облачные IP, которые технически принадлежат не-RU диапазону, но
  обслуживают в том числе российский контент — пойдут через VPN.
- Российские сервисы, разместившие инфраструктуру за пределами RU-диапазонов
  (бывает у части стартапов/CDN) — тоже пойдут через VPN, а не напрямую.
- Точная маршрутизация по конкретным доменам (яндекс напрямую, нетфликс через VPN)
  реализована только на уровне роутера (Keenetic/OpenWrt, см. `router-configs/`)
  через dnsmasq+ipset — там split происходит по факту DNS-резолва домена, а не по IP-диапазону.

### 3. "Отключено логирование" — в рамках того, что контролируете вы сами

Поскольку сервер полностью под вашим управлением, здесь нет "третьей стороны",
которая могла бы вести логи вашего трафика (в отличие от коммерческих VPN-провайдеров).
Но учтите:
- Хостинг-провайдер (Hetzner/Vultr/Timeweb) технически видит объём/направление
  трафика на уровне своей сети (netflow), даже если вы не ведёте логи сами.
  Это стандартное свойство любого VPS/облачного хостинга.
- Если сервер physically/legally скомпрометирован (изъят, взломан, provider
  обязан передать данные по решению суда своей юрисдикции) — угроза выходит за рамки
  того, что решает конфигурация VPN.
- Рекомендация: выбирайте юрисдикцию хостинга осознанно (Финляндия/Германия/Нидерланды
  — страны ЕС/EEA с прозрачным законодательством о защите данных).

### 4. Бэкапы содержат секреты

Архивы `backup.sh` включают приватные ключи WireGuard, `.env` (пароль-хэш панели,
Xray-ключи). Храните их так же осторожно, как сам сервер (шифрованный диск,
ограниченный доступ). Если настроена отправка в Telegram — учтите, что сообщения
доступны всем, у кого есть доступ к соответствующему чату/боту.

### 5. SSH — дополнительно рекомендуется отключить парольную аутентификацию

Этот проект меняет порт SSH (`harden-ssh.sh`) и включает fail2ban, но НЕ отключает
парольный вход автоматически (это не входило в явные требования и рискованно менять
неинтерактивно — можно заблокировать себе доступ, если ключ не настроен). Настоятельно
рекомендуется:
```sh
# Только после того как убедились, что вход по SSH-ключу работает:
sudo sed -i 's/^#\?PasswordAuthentication.*/PasswordAuthentication no/' /etc/ssh/sshd_config
sudo systemctl reload ssh
```

### 6. IPv6 отключен по умолчанию

Если провайдер выдаёт IPv6 и приложение на клиенте использует его напрямую в обход
VPN (IPv6 leak) — это классический вектор утечки реального IP. Поэтому IPv6 отключен
на сервере по умолчанию. Если вам нужен IPv6 — включайте осознанно и убедитесь, что
WireGuard тоже настроен туннелировать IPv6-трафик (потребует дополнительной настройки,
не входящей в этот скрипт).

## Модель угроз (кратко)

Этот проект защищает от: пассивного перехвата трафика на пути провайдер→интернет,
DPI-блокировок по сигнатуре протокола, утечек DNS, случайного доступа к панели
управления извне.

Этот проект НЕ защищает от: репутационных банов IP у стриминговых сервисов,
юридического принуждения хостинг-провайдера, компрометации самого VPS,
трафик-анализа на уровне хостинг-провайдера (netflow), атак на конечное устройство
клиента.
